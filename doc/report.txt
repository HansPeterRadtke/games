Games repo report - 2026-03-27

Current repo state
- branch: `main`
- repo is buildable, smoke-clean, and currently in phase-one modularization
- imported Unity Asset Store content remains local-only and ignored
- repo content is runnable as user `hans`

Phase-one modularization result
1. Central control reduced
- `GameManager` no longer boots the runtime graph by itself
- `SceneBootstrap` now acts as a thin editor/bootstrap adapter that wires scene references into `FpsDemoCompositionRoot`
- runtime service registration is explicit through `com.hpr.composition`

2. New composition system
- new package: `unity/packages/com.hpr.composition`
- core types:
  - `IService`
  - `IServiceResolver`
  - `IServiceRegistry`
  - `IInitializable`
  - `IUpdatableService`
  - `ServiceRegistry`
  - `CompositionRoot`
- headless validation entrypoint:
  - `unity/tools/architecture/run_phase1_headless_validation.sh`

3. Standalone event system
- `com.hpr.eventbus` is now a real standalone event package
- event bus API is now:
  - `IEventBus`
  - `EventBus` (pure C# implementation)
  - `EventManager` (Unity adapter only)
  - `IEventBusSource`
  - `EventBusSourceAdapter`
- gameplay-domain event payloads were moved out of `com.hpr.eventbus`
- gameplay payloads now live in:
  - `unity/packages/com.hpr.core/Runtime/Events/GameplayEvents.cs`

4. Dependency audit output
- generated report:
  - `doc/dependency-audit-phase1.md`
- current phase-one violations are concentrated in `com.hpr.fpsdemo`
- main remaining forbidden patterns:
  - `GameManager` references
  - `SceneBootstrap` ownership of game scene construction
  - parent `MonoBehaviour` fallback lookups in runtime components
- package-level violations outside `fpsdemo` are still present in:
  - `com.hpr.interaction` demo/runtime lookup fallbacks
  - `com.hpr.inventory` demo scene search
  - `com.hpr.stats` parent lookup fallback

5. What is still NOT modular
- `com.hpr.fpsdemo` is still too large and still owns:
  - `GameManager`
  - `SceneBootstrap`
  - `GameStateValidator`
  - game-specific quest/dialogue/menu flow
  - scene-specific composition and smoke flow
- runtime components in `fpsdemo` still rely on scene-parent service discovery in several places
- `com.hpr.core` currently carries FPS-domain event payloads; the event transport is modular, but the domain model is still shared through core rather than fully split into separate game-domain packages

6. Ranked next extraction targets
1. remove parent `MonoBehaviour` service lookups from `fpsdemo` runtime
2. extract input/runtime wiring out of `fpsdemo` and onto explicit composition-only registration
3. move reusable stats runtime out of `fpsdemo` completely
4. move weapon runtime execution out of `fpsdemo`
5. move AI runtime behavior out of `fpsdemo`
6. shrink `SceneBootstrap` until only generic/setup-safe bootstrap logic remains
